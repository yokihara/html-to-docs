# html-to-docs Design

## Summary

`html-to-docs` is a Chrome extension for moving AI agent-generated HTML reports into team documentation tools while preserving both editability and the visual hierarchy of the original report.

The first target is Confluence. Notion and Obsidian are planned follow-up targets, but they do not define MVP quality. The initial product is clipboard-first: users copy a converted `text/html` payload from the extension, then paste it into the Confluence editor.

The product starts as a separate repository because it has independent product boundaries: browser extension packaging, an HTML conversion engine, licensing, and future agent workflow/team features.

## Product Positioning

`html-to-docs` converts AI agent output into team docs.

The first version should solve this user moment:

1. A user has an HTML report generated by Codex, Claude, ChatGPT, or another agent.
2. The report looks good in a browser.
3. The user wants to move it into Confluence without manually rebuilding headings, tables, callouts, cards, badges, colors, and code blocks.
4. The user clicks `Copy for Confluence` in a Chrome extension.
5. The converted content is copied to the clipboard as `text/html` and `text/plain`.
6. The user pastes into a Confluence page and gets an editable document that still keeps the source report's information hierarchy and enough of its design language.

The product is not a pixel-perfect page importer. It is a practical bridge from browser-rendered HTML reports to editable documentation.

## Goals

- Provide a Chrome extension MVP with current-tab and local `.html` file inputs.
- Produce Confluence-friendly clipboard content.
- Preserve document editability for headings, paragraphs, lists, tables, code blocks, links, images, and callouts.
- Preserve enough visual styling to make reports look intentional after paste.
- Establish a conversion engine that can later support Notion, Obsidian, and other documentation tools.
- Separate Free and Pro modes around conversion quality.
- Provide English and Korean extension UI from the MVP.

## Non-Goals

- Do not create Confluence pages through the Confluence REST API in the MVP.
- Do not require Confluence authentication in the MVP.
- Do not use image capture as the primary output path.
- Do not attempt pixel-perfect preservation.
- Do not build history, batch export, team presets, or agent folder watching in the MVP.
- Do not implement a production payment system in the MVP, though the license boundary should exist.

## Repository Shape

The product should live in a new repository:

```text
html-to-docs/
  apps/
    extension/
  packages/
    converter/
    shared/
  docs/
    superpowers/
      specs/
```

Only the design document exists at this stage. Implementation should start later from a dedicated implementation plan.

## Architecture

### Chrome Extension

`apps/extension` will be a Chrome Manifest V3 extension.

Responsibilities:

- Show a compact popup UI.
- Detect the current tab.
- Let users choose a local `.html` file.
- Let users choose the target, initially Confluence.
- Call the converter package with the selected HTML source and license mode.
- Write converted output to the clipboard as `text/html` and `text/plain`.
- Show concise success and failure states.
- Surface Pro conversion as a locked or mock-gated capability until licensing is implemented.
- Localize all user-facing extension UI strings in English and Korean.

The extension should avoid becoming the conversion engine. It owns browser permissions, source collection, and clipboard UX. Conversion rules belong in `packages/converter`.

### Internationalization

The MVP should include English and Korean UI support.

The extension should use Chrome's standard i18n structure:

```text
apps/extension/
  _locales/
    en/
      messages.json
    ko/
      messages.json
```

The extension manifest, popup UI, settings UI, error messages, warning messages, and upgrade copy should use localization keys instead of hard-coded strings. English should be the default locale.

The converter must not translate user-provided HTML content. It should only preserve and transform the source document structure and styling. Localization applies to the product UI and system messages, not to report contents.

### Content Script

The content script reads the current tab DOM when the popup requests conversion.

Responsibilities:

- Capture the relevant document HTML.
- Prefer the rendered DOM over raw page source when possible.
- Exclude extension UI and browser-only artifacts.
- Return a serialized HTML snapshot to the extension runtime.

The content script should report actionable errors when a page cannot be read because of Chrome permissions, restricted schemes, sandboxing, or local file settings.

### Converter Package

`packages/converter` is the core product asset.

Input:

- Raw HTML string, or
- Serialized DOM snapshot

Output:

- `text/html` clipboard payload
- `text/plain` fallback
- warnings about removed or simplified content

Initial modes:

- `confluence-basic`
- `confluence-pro`

Basic mode prioritizes stable paste behavior. Pro mode differentiates the paid product by preserving more report design.

### Shared Package

`packages/shared` contains cross-package types and constants.

Examples:

- target platform identifiers
- conversion mode identifiers
- license status
- conversion options
- warning/error codes
- telemetry event names, if telemetry is later added

The MVP can use a mock license provider, but the interface should already model `free`, `pro`, and `team` states.

## Conversion Strategy

The converter should normalize HTML into content that Confluence's editor can accept through paste.

The engine should remove or ignore document-hostile content:

- `script`
- unsafe event handlers
- tracking pixels
- forms
- unsupported iframes
- external behavior dependencies
- styles that are likely to break paste or editing

The engine should preserve semantic content:

- headings
- paragraphs
- links
- ordered and unordered lists
- tables
- blockquotes
- code blocks
- inline code
- images where safe and supported

The converter should turn common report UI patterns into paste-friendly equivalents:

- cards become bordered content blocks
- badges become inline pill spans
- callouts become colored blockquotes or bordered blocks
- code panels become `pre > code`
- table headers keep useful background and weight
- grid and flex layouts collapse into readable vertical sections when needed

### Basic Mode

Basic mode should be reliable and conservative.

It should:

- remove unsafe or unsupported elements
- preserve core semantic structure
- simplify complex layout
- produce good plain-text fallback
- avoid fragile inline styling

Basic mode is the free value demonstration. It should work well enough that users trust the product.

### Pro Mode

Pro mode should preserve report design more aggressively.

It should preserve this initial allowlist of inline styles when the target paste behavior remains stable:

- `color`
- `background-color`
- `border`
- `border-radius`
- `padding`
- `margin`
- `font-weight`
- `font-style`
- `text-align`
- `width`
- `max-width`
- `vertical-align`

It should avoid styles that are likely to create broken paste results:

- complex positioning
- animations
- transforms
- viewport-sized fixed layouts
- external font dependencies
- unsupported grid/flex behavior when Confluence cannot preserve it

Pro mode should be marketed as `Preserve report design`, not as exact cloning.

## UX

The popup should be task-oriented rather than dashboard-like.

Primary actions:

- `Copy for Confluence`
- `Open HTML file`

Secondary states:

- current tab detected
- local file selected
- Basic/Pro mode indicator
- upgrade link when Pro is locked
- concise warnings after conversion

Success message:

> Copied. Paste into the Confluence editor.

Failure messages should be actionable:

- Page cannot be read because of Chrome permissions.
- Local file access is disabled for this extension.
- HTML is too large to convert.
- Some unsupported elements were removed.
- Clipboard write failed.

The extension should not require users to understand the converter internals. It should make the common path feel like one button.

All visible extension UI should ship in English and Korean. The product can choose the browser locale automatically, with a later settings override if needed.

## Monetization

The MVP should be treated as the first publicly distributable free commercial version, not just an internal prototype. Users should be able to install it and use the Free feature set without payment.

The MVP may show Pro as a locked upgrade path, but it should not depend on production billing. Real checkout, account management, and license enforcement belong to the later licensing phase. Until then, Pro can be represented through disabled controls, a waitlist/upgrade link, or a mock license flag for development.

The first paywall should be conversion quality.

Free:

- Basic Confluence copy
- current tab input
- local HTML input
- conservative semantic conversion

Pro:

- stronger design preservation
- card, badge, callout, table, code-block, color, and spacing preservation
- better plain-text fallback
- warnings that explain what changed

Future higher tiers:

- agent output folder watching
- conversion history
- batch export
- team presets
- shared style templates
- organization defaults
- local-processing policy controls

Chrome Web Store payments should not be the assumed billing system. The extension should later use an external checkout and a license API. Stripe, Paddle, Lemon Squeezy, Gumroad, or ExtensionPay can be evaluated during implementation planning.

## Testing Strategy

The biggest product risk is actual Confluence paste behavior. Tests should cover both converter correctness and real paste outcomes.

Converter tests:

- fixture HTML input
- Basic output snapshots
- Pro output snapshots
- text fallback snapshots
- warning/error snapshots
- sanitizer tests for hostile HTML

Extension tests:

- current tab extraction
- local file reading
- clipboard payload creation
- permission failures
- large input failure
- mode selection

Manual/semiautomated QA:

- maintain a set of representative agent report fixtures
- paste each fixture into a Confluence test page
- record screenshots and checklist results
- note where Confluence strips styles
- feed findings back into converter rules

The first QA checklist should verify:

- no overlapping text
- readable section hierarchy
- editable headings and paragraphs
- editable tables
- readable code blocks
- recognizable callouts/cards/badges
- acceptable plain-text fallback

## Roadmap

### Phase 1: MVP Skeleton

- scaffold monorepo
- create Chrome extension shell
- create converter package shell
- support current tab HTML extraction
- support local `.html` input
- implement Basic and Pro mode interfaces
- write clipboard payloads
- add initial converter fixtures

### Phase 2: Confluence Quality

- expand fixture library with real agent reports
- harden sanitizer
- improve card, badge, callout, table, and code-block conversions
- build manual Confluence paste checklist
- document known Confluence paste limitations

### Phase 3: Licensing

- choose billing provider
- add account/checkout flow
- add license API
- unlock Pro mode
- add extension settings for license status

### Phase 4: Additional Targets

- add Notion output mode
- add Obsidian output mode
- revisit target-specific styling and markdown/HTML tradeoffs

### Phase 5: Workflow And Team Features

- agent output folder watching
- conversion history
- batch export
- team presets
- shared templates
- organization defaults

## Open Decisions For Implementation Planning

- Which frontend stack should the extension popup use: plain TypeScript, React, or another lightweight UI layer?
- Which package manager and monorepo tool should be used?
- Should converter tests run in Node with an HTML parser, in a browser DOM environment, or both?
- Which billing provider best matches Chrome extension licensing?
- What is the first set of representative agent report fixtures?
- Should users be able to override the detected UI language in settings, or should the MVP rely only on browser locale?

## Approval

This design was approved in conversation on 2026-06-16 with the following direction:

- start with a Chrome extension
- prioritize Confluence
- balance editability and visual preservation
- use Free for basic conversion
- use Pro as the first paywall for higher-quality design preservation
- reserve agent workflow and team preset features for higher tiers
- keep the product in a separate `html-to-docs` project
- include English and Korean as default extension UI languages

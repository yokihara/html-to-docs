# html-to-docs Design

## Summary

`html-to-docs` helps move AI agent output into team documentation tools as native, editable documents.

The original clipboard-first idea remains useful as a fallback, but testing showed that Confluence Cloud strips or rewrites too much pasted HTML/CSS for "prettier paste" to be a strong Pro differentiator. The revised MVP should focus on extracting a platform-neutral document intent model from agent output, then producing Confluence-native instructions or MCP-ready output before the final document is created.

The first target is still Confluence because the user pain is clearest there. Notion and Obsidian should remain first-class future targets through the same document intent model, not through Confluence-specific paste rules.

## Product Positioning

`html-to-docs` converts AI agent output into docs-native artifacts.

The revised first version should solve this user moment:

1. A user has an HTML report generated by Codex, Claude, ChatGPT, or another agent.
2. The report looks good in a browser, but paste fidelity into Confluence is poor.
3. The user wants the final result to become an editable Confluence page or section without manually rebuilding headings, tables, callouts, status badges, and code blocks.
4. The tool extracts the report's document intent: sections, callouts, status markers, tables, code blocks, lists, and emphasis.
5. If Atlassian MCP is available, the tool prepares or drives a Confluence-native create/update workflow.
6. If MCP is not available, the tool falls back to `Copy Clean`: a readable, conservative clipboard payload.

The product is not a pixel-perfect page importer. It is a bridge from agent output to documentation-native structure. HTML paste is a rescue path, not the long-term premium value.

## Goals

- Provide a Chrome extension MVP with current-tab and local `.html` file inputs.
- Extract a platform-neutral document intent model from agent-generated HTML reports.
- Generate a Confluence-native body and MCP-ready action plan from that model.
- Keep `Copy Clean` as the free fallback path for readable clipboard paste.
- Preserve document editability for headings, paragraphs, lists, tables, code blocks, links, images, callouts, and status-like labels.
- Establish a renderer architecture that can later support Notion, Obsidian, and other documentation tools.
- Separate Free and Pro around workflow capability, not small paste-style differences.
- Provide English and Korean extension UI from the MVP.

## Non-Goals

- Do not chase pixel-perfect HTML/CSS paste fidelity as the main product promise.
- Do not make "prettier paste" the primary Pro differentiator.
- Do not require Confluence authentication for the free fallback path.
- Do not use image capture as the primary output path.
- Do not build history, batch export, team presets, or agent folder watching in the MVP.
- Do not implement a production payment system in the MVP, though the license boundary should exist.

## Repository Shape

The product should live in a new repository:

```text
html-to-docs/
  apps/
    extension/
  packages/
    converter/
    shared/
  docs/
    superpowers/
      specs/
```

The initial extension skeleton exists, but the next implementation phase should refactor the product around the revised document intent architecture before further paste-fidelity work.

## Architecture

### Chrome Extension

`apps/extension` will be a Chrome Manifest V3 extension.

Responsibilities:

- Show a compact popup UI.
- Detect the current tab.
- Let users choose a local `.html` file.
- Let users choose the target, initially Confluence.
- Call the converter package to extract a document intent model.
- Offer `Copy Clean` for conservative clipboard output.
- Offer `Native Doc Mode` for direct Confluence MCP execution by an agent.
- Show concise success and failure states.
- Surface Pro/MCP workflow as a locked, preview, or mock-gated capability until licensing is implemented.
- Localize all user-facing extension UI strings in English and Korean.

The extension should avoid becoming the document intelligence layer. It owns browser permissions, source collection, clipboard UX, and handoff UX. Extraction and target rendering rules belong in `packages/converter`.

### Internationalization

The MVP should include English and Korean UI support.

The extension should use Chrome's standard i18n structure:

```text
apps/extension/
  _locales/
    en/
      messages.json
    ko/
      messages.json
```

The extension manifest, popup UI, settings UI, error messages, warning messages, and upgrade copy should use localization keys instead of hard-coded strings. English should be the default locale.

The converter must not translate user-provided HTML content. It should only preserve and transform the source document structure and styling. Localization applies to the product UI and system messages, not to report contents.

### Content Script

The content script reads the current tab DOM when the popup requests conversion.

Responsibilities:

- Capture the relevant document HTML.
- Prefer the rendered DOM over raw page source when possible.
- Exclude extension UI and browser-only artifacts.
- Return a serialized HTML snapshot to the extension runtime.

The content script should report actionable errors when a page cannot be read because of Chrome permissions, restricted schemes, sandboxing, or local file settings.

### Converter Package

`packages/converter` is the core product asset.

Input:

- Raw HTML string, or
- Serialized DOM snapshot

Output:

- platform-neutral document intent model
- `Copy Clean` clipboard payload
- Confluence-native body and MCP-ready action payload
- warnings about ambiguous or simplified content

Initial responsibilities:

- sanitize and parse hostile or messy HTML
- identify document structure
- classify report components such as callouts, status labels, cards, tables, and code blocks
- render the model for Confluence fallback paste
- render the model as an MCP-ready Confluence creation/update instruction

The package should be split internally into core extraction and target adapters. Confluence-specific behavior must not become the permanent core model.

### Shared Package

`packages/shared` contains cross-package types and constants.

Examples:

- target platform identifiers
- conversion mode identifiers
- license status
- conversion options
- warning/error codes
- telemetry event names, if telemetry is later added

The MVP can use a mock license provider, but the interface should already model `free`, `pro`, and `team` states.

## Conversion Strategy

The converter should first extract document intent, then render that intent for a target.

### Document Intent Model

The model should describe what the report means, not how the browser painted it.

Initial block types:

- document title
- section
- paragraph
- ordered and unordered list
- table
- code block
- inline code
- callout with tone
- status badge or label with tone
- card or phase block
- quote
- image reference when safe

The model should preserve hierarchy, ordering, text, links, table structure, code, and semantic emphasis. Color is represented as intent or tone, not raw CSS.

### Sanitization

The engine should remove or ignore document-hostile content before extraction:

- `script`
- unsafe event handlers
- tracking pixels
- forms
- unsupported iframes
- external behavior dependencies
- styles that are only meaningful in a browser layout

### Target Rendering

The same model should support multiple target renderers.

Confluence `Copy Clean` renderer:

- readable `text/html` and `text/plain`
- conservative tables, headings, lists, code blocks, and callouts
- no promise of visual fidelity

Confluence `Native Doc Mode` renderer:

- execution payload for an agent with Atlassian MCP access
- target page title, parent/page selection hints, and structured blocks
- explicit mapping from callout/status/table/code intent to Confluence-native capabilities
- update/create safety checklist before writing

Future renderers:

- Notion block renderer
- Obsidian Markdown + safe HTML renderer
- Google Docs or Slack Canvas renderer if user demand exists

### Free Mode

Free mode should be reliable and conservative.

It should:

- extract document intent locally
- generate `Copy Clean` clipboard output
- preserve core semantic structure
- simplify complex layout
- produce good plain-text fallback
- explain when the output is a fallback rather than a native document

Free mode is the value demonstration. It should reduce manual cleanup, but it should not be marketed as high-fidelity design preservation.

### Pro / Native Doc Mode

Pro should focus on native workflow, not marginally prettier paste.

It should:

- generate a Confluence-native document plan from the intent model
- prepare an Atlassian MCP action payload for direct tool execution
- optionally create or update a Confluence page when the user has MCP connected and confirms the action
- preserve callout/status/table/code intent through Confluence-native structures where possible
- show a preview of the document outline and target operations before execution

Pro should be marketed as `Native Doc Mode` or `Agent-to-Docs Workflow`, not as `Preserve report design`.

## UX

The popup should be task-oriented rather than dashboard-like.

Primary actions:

- `Copy Clean`
- `Prepare Native Doc`
- `Open HTML file`

Secondary states:

- current tab detected
- local file selected
- fallback/native mode indicator
- MCP connection status, if available
- upgrade link when Native Doc Mode is locked
- concise warnings after extraction or rendering

Copy fallback success message:

> Copied. Paste into the Confluence editor.

Native doc success message:

> Native doc draft prepared. Review before creating or updating Confluence.

Failure messages should be actionable:

- Page cannot be read because of Chrome permissions.
- Local file access is disabled for this extension.
- HTML is too large to convert.
- Some unsupported elements were removed.
- Clipboard write failed.
- Atlassian MCP is not connected.
- Target Confluence page could not be resolved.
- Native document generation needs user review before write.

The extension should not require users to understand the converter internals. It should make the fallback path feel like one button and the native path feel like a guided handoff.

All visible extension UI should ship in English and Korean. The product can choose the browser locale automatically, with a later settings override if needed.

## Monetization

The MVP should be treated as the first publicly distributable free commercial version, not just an internal prototype. Users should be able to install it and use the Free feature set without payment.

The MVP may show Native Doc Mode as a locked or preview upgrade path, but it should not depend on production billing. Real checkout, account management, and license enforcement belong to the later licensing phase. Until then, Pro can be represented through disabled controls, a waitlist/upgrade link, or a mock license flag for development.

The first paywall should be native workflow capability.

Free:

- Copy Clean fallback
- current tab input
- local HTML input
- local document intent extraction
- conservative semantic clipboard conversion

Pro:

- Native Doc Mode
- Confluence MCP action payload generation
- guided Confluence create/update flow when MCP is available
- richer document intent inspection
- warnings that explain ambiguous blocks and target limitations

Future higher tiers:

- agent output folder watching
- conversion history
- batch export
- team presets
- shared style templates
- organization defaults
- local-processing policy controls
- approved target adapters for multiple documentation tools

Chrome Web Store payments should not be the assumed billing system. The extension should later use an external checkout and a license API. Stripe, Paddle, Lemon Squeezy, Gumroad, or ExtensionPay can be evaluated during implementation planning.

## Testing Strategy

The biggest product risk is whether the extracted document intent produces useful native documentation. Paste behavior remains a fallback risk, but it is no longer the core Pro promise.

Converter tests:

- fixture HTML input
- document intent model snapshots
- Copy Clean output snapshots
- Confluence native body/action snapshots
- text fallback snapshots
- warning/error snapshots
- sanitizer tests for hostile HTML

Extension tests:

- current tab extraction
- local file reading
- clipboard payload creation
- Native Doc Mode handoff UI
- MCP connection unavailable state
- permission failures
- large input failure
- mode selection

Manual/semiautomated QA:

- maintain a set of representative agent report fixtures
- compare source HTML, extracted document intent, Copy Clean output, and Native Doc plan
- for Copy Clean, paste each fixture into a Confluence test page and record limitations
- for Native Doc Mode, create/update a test Confluence page through MCP or a mocked MCP payload and review the resulting structure
- feed findings back into extraction and target renderer rules

The first QA checklist should verify:

- readable section hierarchy
- editable headings and paragraphs
- editable tables
- readable code blocks
- recognizable callouts/cards/status labels
- acceptable plain-text fallback
- native document outline matches source intent
- target write operation is reviewed before execution

## Roadmap

### Phase 1: Current Extension Skeleton

- scaffold monorepo
- create Chrome extension shell
- create converter package shell
- support current tab HTML extraction
- support local `.html` input
- implement Copy Clean and Pro preview experiments
- write clipboard payloads
- add initial converter fixtures

### Phase 2: Document Intent Refactor

- introduce document intent model types
- refactor converter into extraction plus target renderers
- create `copy-clean` renderer
- create Confluence native body/action renderer
- preserve current clipboard behavior as fallback
- add model snapshot tests

### Phase 3: Atlassian MCP Native Doc Mode

- detect whether Atlassian MCP/Rovo workflow is available where possible
- generate MCP-ready actions for Confluence create/update
- add review-before-write UX
- support title, parent/page hints, and page update safety
- document MCP limitations and required user confirmation

### Phase 4: Confluence Quality

- expand fixture library with real agent reports
- harden sanitizer
- improve callout, status label, table, code-block, and section extraction
- build native Confluence QA checklist
- document known Confluence paste and MCP limitations

### Phase 5: Licensing

- choose billing provider
- add account/checkout flow
- add license API
- unlock Native Doc Mode
- add extension settings for license status

### Phase 6: Additional Targets

- add Notion output mode
- add Obsidian output mode
- revisit target-specific native block and markdown/HTML tradeoffs

### Phase 7: Workflow And Team Features

- agent output folder watching
- conversion history
- batch export
- team presets
- shared templates
- organization defaults

## Open Decisions For Implementation Planning

- Which frontend stack should the extension popup use: plain TypeScript, React, or another lightweight UI layer?
- Which package manager and monorepo tool should be used?
- What should the first document intent model schema include?
- Should extractor tests run in Node with an HTML parser, in a browser DOM environment, or both?
- What should the initial Atlassian MCP handoff format look like?
- Which billing provider best matches Chrome extension licensing?
- What is the first set of representative agent report fixtures?
- Should users be able to override the detected UI language in settings, or should the MVP rely only on browser locale?

## Approval

This design was approved in conversation on 2026-06-16 with the following direction:

- start with a Chrome extension
- prioritize Confluence
- treat paste conversion as a fallback, not the premium product promise
- use Free for Copy Clean fallback and local intent extraction
- use Pro/Native Doc Mode for Atlassian MCP or native Confluence workflow
- preserve multi-target future through a platform-neutral document intent model
- keep the product in a separate `html-to-docs` project
- include English and Korean as default extension UI languages
